## Major refactors

### 1. AppContext extraction

**Intent:** Isolate UI calls from engine code (effects, vsthost, transitions, keyframes, loaders, project model) to support headless operation.

**Files affected:**

| 1.x | 2.0.x | Migration |
|-----|-------|-----------|
| `src/effects/effect.cpp` (direct QMessageBox, file dialog calls) | `src/effects/effect.cpp` (calls to `app_ctx->`) | Engine now queries AppContext instead of QWidget directly |
| `src/effects/vsthost.cpp` (same) | `src/effects/vsthost.cpp` | Same pattern |
| `src/engine/transition/transition.cpp` | `src/engine/transition/transition.cpp` | Null guard: `if (app_ctx)` checks added |
| (keyframe/loadthread/projectmodel UI calls) | `src/core/appcontext.h` methods | New methods: `file_dialog()`, `clearViewerMedia()`, `getMainWindow()`, `clearGraphEditorIfRow()` |

**Backport implication:** When fixing crashes in engine code related to `QMessageBox` or file dialogs, check if the 1.x call is now routed through AppContext in 2.0.x. Lines calling `QMessageBox::warning()` directly in 1.x may need to call `app_ctx->...()` instead.

**Implementation location:** `src/ui/appcontextimpl.cpp` (concrete implementations) and `src/core/appcontext.h` (interface).

### 2. ProjectIO extraction

**Intent:** Factor out project state management (save, load, autorecovery, signal emission) from monolithic `Global` class to support headless project manipulation.

**Files affected:**

| 1.x | 2.0.x | Details |
|-----|-------|---------|
| `src/global/global.h` (~860 lines) | Split into `src/global/global.h` + `src/global/projectio.h` | ~860 ã 797 + 68 lines |
| `src/global/global.cpp` (full impl) | `src/global/projectio.cpp` (extracted methods) | `SaveProject()`, `LoadProject()`, `SaveAutoRecover()`, autorecovery signals moved |
| (project lifecycle methods scattered) | Cohesive `ProjectIO` class | New class: `ProjectIO` in `global/projectio.h` |

**Backport implication:** If a 1.x fix touches `Global::SaveProject()` or `LoadProject()`, look for the method in `ProjectIO` in 2.0.x. Initialization order may differ; check 2.0.x `MainWindow` constructor.

**Verify:** `git diff main..2.0.x -- src/global/`

### 3. EffectField split (data vs widget)

**Intent:** Decouple effect parameter storage from UI representation to allow headless effect evaluation.

**Files affected:**

| 1.x | 2.0.x | Purpose |
|-----|-------|---------|
| `src/effects/effectfield.h` (mixed data+UI) | `src/effects/effectfield.h` (reduced to interface) | Now holds only the core `EffectField` base interface |
| `src/effects/fields/*.h/cpp` (14 files with UI) | Moved to `src/effects/ui/*.h/cpp` | Widget subclasses: `BoolFieldWidget`, `DoubleFieldWidget`, `ColorFieldWidget`, `StringFieldWidget`, `ComboFieldWidget`, `FileFieldWidget`, `FontFieldWidget`, `ButtonFieldWidget`, `LabelFieldWidget` |

**Data class mapping:**

- Text field data: lives in `EffectField`, UI rendering in `StringFieldWidget`
- Double/slider data: `EffectField`, UI in `DoubleFieldWidget`
- Color data: `EffectField`, UI in `ColorFieldWidget`
- File picker data: `EffectField`, UI in `FileFieldWidget`

**Backport implication:** Fixes to field rendering (e.g., color picker layout, double-field precision) now touch `src/effects/ui/*FieldWidget.cpp`. Fixes to field value storage/keyframe handling touch `src/effects/effectfield.cpp` or `src/effects/effectrow.cpp`.

**Total stat:** 48 files changed: 892 insertions, 727 deletions. Largest refactor by file count.

### 4. amber-engine OBJECT library

**Intent:** Separate headless rendering engine (no Qt UI dependencies) into reusable CMake OBJECT library for test harness.

**CMake target definition:**

`src/CMakeLists.txt` now defines:
```
add_library(amber-engine OBJECT ...)
```

**What it includes:**
- `src/engine/` ã rendering, caching, effects, transitions, keyframes
- `src/project/` ã sequence, clip, media, footage metadata
- `src/core/` ã utility functions, color labels, math, display types
- `src/effects/internal/` ã Transform, Text, Solid, Noise, etc.
- `src/rendering/` ã audio, video codecs, RHI pipeline

**What it excludes:**
- `src/ui/` (except AppContext interface in `src/core/appcontext.h`)
- `src/dialogs/`
- `src/panels/`
- All Qt GUI classes (QMainWindow, QDockWidget, etc.)

**Boundary check:** Link time will fail if you accidentally include a UI header in engine code. Used by:
- `amber-gui` (main executable, links object lib + UI layers)
- `test_rendering` (headless test binary)
- `test_rhi_smoke` (QRhi initialization test)
- `test_projectio` (project I/O regression tests)

**Backport implication:** If a 1.x fix in `src/engine/` or `src/effects/` includes a `#include` from `src/ui/`, it will fail in 2.0.x. Remove or route through AppContext instead.

## Save Frame cherry-pick example (db5987332)

### The fix

The commit adds absolute path anchoring to the Save Frame dialog to prevent relative paths from resolving against the project directory (common on XFCE/GTK).

### 1.x target

**File:** `src/ui/viewerwidget.cpp`  
**Method:** `ViewerWidget::save_frame()`  
**Lines (~192-213):** 
- Add includes: `QFileInfo`, `QStandardPaths`
- Set dialog directory to Pictures or saved last location
- Convert selected file to absolute path
- Cache last directory for next invocation

**Header change:** Add `QString save_frame_last_dir_` member to `src/ui/viewerwidget.h`

### 2.0.x target

**Same files and method exist.** However:

1. **Include list differs:** 2.0.x viewerwidget already includes many Qt headers but NOT `QFileInfo` or `QStandardPaths` by default (due to CC reduction extracting paint code). Check existing includes in line 20-50.

2. **Method location unchanged:** `src/ui/viewerwidget.cpp` line ~195 still contains `save_frame()`.

3. **Existing structure:** The method is already cleaned up in 2.0.x (no extra includes). The patch applies cleanly to the same location with minimal conflict.

### Why cherry-pick reported conflict

If you attempted `git cherry-pick db5987332 2.0.x` and got a conflict on `src/ui/viewerwidget.cpp`, it likely due to:
- Include section reordered (2.0.x moved/removed some headers)
- Context lines mismatch (blank lines added during CC refactoring)

### Suggested manual port

1. In 2.0.x `src/ui/viewerwidget.h`, add member: `QString save_frame_last_dir_;` (around line 147 where other members live)
2. In 2.0.x `src/ui/viewerwidget.cpp`, locate `save_frame()` method (~line 195)
3. Add includes if missing: `#include <QFileInfo>` and `#include <QStandardPaths>`
4. Apply the diff logic (set directory, make path absolute, cache location)
5. Verify no conflicts with nearby methods

The logic itself is independent of the refactors; only the surrounding line numbers may shift.

## Backport checklist for 1.x ã 2.0.x

1. **Identify the 1.x file(s) the fix touches.**
   - Example: Fix in `src/effects/effect.cpp` ã check if AppContext extraction affects it.

2. **Cross-reference this document for new locations.**
   - AppContext calls? Look in `src/core/appcontext.h` and `src/ui/appcontextimpl.cpp`.
   - ProjectIO methods? Check `src/global/projectio.h`.
   - Field UI rendering? Check `src/effects/ui/*FieldWidget.cpp`.
   - Timeline logic? Verify function not split into helpers.
   - Pure utility (blur, colorlabel)? New location in `src/core/`.

3. **Check for include changes.**
   - `#include "ui/blur.h"` ã `#include "core/blur.h"`
   - `#include "timeline/marker.h"` ã `#include "core/marker.h"`
   - `OLIVE_` macros ã `AMBER_` macros

4. **If targeting engine code, verify no UI includes were added.**
   - This will break 2.0.x's amber-engine OBJECT library boundary.
   - Route through AppContext if UI interaction is needed.

5. **Run local tests:**
   - `ctest --verbose` ã verify test_rhi_smoke, test_rendering, test_projectio pass
   - Manual smoke test: open a project, play timeline, save frame

6. **For timeline/cyclomatic refactors, search by function name, not line number.**
   - Large functions may have been split. Use Ctrl+F for the function, not git blame line numbers.
